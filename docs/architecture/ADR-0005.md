# ADR 0005: Accessibility Smoke Tests

## Status

Proposed for the testing roadmap. Implementation is deferred.

## Context

Transport Studio is a dense editor-style UI with a project tree, mode navigation, viewport, inspector, run panel, and many icon-like controls.

The current frontend test suite verifies rendering, editor workflows, and browser-level stability. It does not yet explicitly verify accessibility semantics. That leaves room for regressions where the UI still looks correct but loses usable roles, labels, heading structure, keyboard reachability, or ARIA state.

Because the project tree and editor shell already rely heavily on semantic roles in tests, accessibility smoke tests are a natural extension of the Playwright layer.

## Decision

Add accessibility smoke tests using `@axe-core/playwright` on top of Playwright.

The tests should scan the main Studio shell and focused editor regions:

- app shell
- project tree
- viewport-adjacent controls
- inspector
- run panel

These tests should run in a browser but remain smoke-level. They should catch obvious accessibility regressions without becoming a full manual accessibility audit.

## Scope

The first accessibility smoke suite should verify:

- the main Studio shell has no serious or critical axe violations
- the project tree exposes stable tree, group, item, button, and label semantics
- mode navigation remains discoverable by accessible name
- inspector headings and form labels remain usable
- run panel tabs and actions remain reachable and labeled
- disabled, locked, hidden, selected, and active states have meaningful semantic expression where the UI exposes them

The suite should prefer targeted scans over whole-page scans when a whole-page scan creates noisy results from unrelated visual or canvas content.

## Severity policy

The first pass should fail on serious and critical axe violations.

Lower-severity violations may be reported but should not automatically block the suite until the team has reviewed whether they are meaningful for this UI.

Known exceptions must be explicit, narrow, and tied to a documented reason. Broad axe-rule suppression is not allowed.

## Relationship to Playwright tests

Accessibility smoke tests should share the Playwright harness where possible:

```txt
Start Vite app
  -> load Studio shell
  -> navigate to region or state
  -> run axe scan
  -> assert no unexpected serious or critical violations
```

They should not duplicate every e2e workflow. The goal is semantic regression coverage, not another behavior suite.

## Canvas limitation

The viewport canvas is not itself a complete accessibility surface.

The accessibility obligation for the canvas region is to provide surrounding semantic controls, status summaries, and diagnostics that expose the relevant editor and run state outside the pixels.

This ADR does not require making sampled tracks or 3D/2D render objects directly screen-reader navigable.

## Non-goals

- Do not implement a full WCAG compliance program in this slice.
- Do not add snapshot-based visual accessibility checks.
- Do not require all canvas-rendered transport visuals to become DOM nodes.
- Do not test every project-tree permutation through axe.
- Do not make accessibility smoke tests part of mutation testing.

## Consequences

- The UI gains a low-cost guardrail against losing roles, labels, and state semantics.
- Some existing controls may need better accessible names before the suite can be made blocking.
- The test suite may expose design questions about icon-only controls, disabled actions, and tree-item state.
- Accessibility regressions become easier to catch before they harden into interaction patterns.

## Required follow-up tests

When implemented, add smoke tests for:

- clean shell scan after initial app load
- project tree scan with default entities
- project tree scan after hidden and locked states are applied
- inspector scan for a selected geometry entity
- run panel scan after a toy photon run
- diagnostic output scan for the native bridge-unavailable path
