# ADR 0004: Property-Based Unit Tests for Pure Logic

## Status

Proposed for the testing roadmap. Implementation is deferred.

## Context

Transport Studio already has fast Vitest unit coverage, browser-level Playwright flows, Playwright-driven fuzz coverage, and a separate Stryker mutation-testing path.

The browser fuzz layer is useful for editor workflow stability, but it is too expensive and indirect for many pure logic defects. Mutation testing has also shown that several high-signal logic surfaces can survive example-based tests. Those survivors are better attacked with focused property-based unit tests than with larger browser flows.

The most useful next step is to use `fast-check` outside the browser against deterministic domain and editor functions.

## Decision

Add property-based unit tests for pure TypeScript logic using `fast-check` in the Vitest unit-test layer.

These tests should live next to the logic they exercise and run without Playwright, DOM automation, Vite dev servers, or native/Tauri runtime dependencies.

Property tests should be considered part of the fast unit suite when their runtime stays small. If a property becomes expensive, it should be explicitly isolated behind a separate script rather than silently slowing the default unit path.

## Initial targets

The first property-test targets should be functions with clear invariants and high mutation-testing value:

- material fraction normalization
- material readiness and diagnostic summaries
- geometry validation and analytic volume estimates
- source direction normalization
- tally target validation
- project mutation invariants
- selection reducers
- visibility and lock reducers
- stale-result marker reducers

## Property style

Property tests should assert general truths rather than duplicate current examples with random values.

Good properties include:

- normalized valid fractions sum to one within a documented tolerance
- invalid or zero-total fractions are preserved or rejected according to the domain contract
- source direction normalization never creates `NaN` or infinite components
- a zero direction remains invalid rather than being silently repaired
- geometry validation is stable under entity ordering when IDs remain the same
- project mutations preserve unrelated entities
- duplicate operations create fresh IDs and do not mutate the original entity
- delete operations remove the target and leave unrelated IDs intact
- visibility toggles do not change compile-inclusion state unless that behavior is explicitly added later
- selection gracefully handles missing, hidden, locked, and deleted entities

## Test data boundaries

Arbitraries should be deliberately bounded.

They should avoid generating unrealistic data so broad that failures become noise. Prefer small numeric ranges, valid ID shapes, simple entity collections, and explicit invalid-case generators.

When a generated value represents a physically impossible state, the test should make that precondition visible instead of letting impossible data leak into unrelated assertions.

## Replay and determinism

Failures must include enough information to reproduce the generated case.

The implementation should preserve `fast-check` seed and path output. If custom wrappers are added, they must not swallow the seed, path, shrunk counterexample, or property label.

No failure should be dismissed as random. A failure means either the app has a bug, the property is wrong, or the generator permits invalid states without a clear precondition.

## Relationship to mutation testing

Property-based unit tests should be used to reduce mutation survivors in pure logic before increasing Stryker thresholds.

The preferred workflow is:

```txt
Run mutation baseline
  -> inspect surviving logic mutants
  -> add or refine property tests
  -> rerun targeted unit tests
  -> rerun mutation testing separately
  -> raise thresholds only after repeated baseline stability
```

Mutation testing remains separate from the default test path. Property tests may join the default unit path only while they remain fast and deterministic.

## Non-goals

- Do not add browser automation in this slice.
- Do not add new UI flows.
- Do not use property tests for visual layout assertions.
- Do not generate arbitrary React component trees.
- Do not replace example-based tests where a named regression is clearer.
- Do not run Stryker as part of this ADR.

## Consequences

- Pure domain/editor logic receives broader input coverage without paying browser-test cost.
- Mutation survivors become easier to address with small, focused tests.
- Test failures may expose ambiguous contracts that need domain clarification before implementation changes.
- Poorly bounded arbitraries can create noisy tests, so generators must be designed as carefully as assertions.
- Some example-based tests should remain as readable contract documentation even after property tests exist.

## Required follow-up tests

When implemented, add focused property tests for:

- valid and invalid material fraction normalization
- source direction normalization and invalid zero directions
- geometry validation stability across simple generated scenes
- project mutation preservation of unrelated entities
- create, duplicate, delete, visibility, and lock invariants
- selection behavior after deletion or missing references
- stale-result marker behavior after editor-scene changes
