# ADR 0006: TS/Rust Bridge Contract Fixtures

## Status

Accepted and implemented by the Native Execution Contract v2 conformance corpus.

## Context

Transport Studio has a browser-first React frontend, a TypeScript transport/domain layer, and a Tauri/Rust native path for future transport execution.

The frontend can run without the native host, and native bridge calls must degrade cleanly to a bridge-unavailable diagnostic in browser-only development. Full Tauri desktop automation would be expensive for the current stage, but the TypeScript/Rust boundary still needs protection.

The most valuable next step is to pin the payload and event contracts exchanged by the native photon bridge with shared golden fixtures.

## Decision

Keep one shared TS/Rust conformance corpus for the native photon path before adding full Tauri e2e automation.

The fixtures should define representative request payloads, backend event streams, diagnostics, and bridge-unavailable responses. TypeScript tests and Rust tests should both consume the same fixture data or fixture schema so that bridge drift is caught at the contract boundary.

The runtime-neutral corpus lives at `fixtures/contracts/native-execution-v2.json`. It covers the request envelope, every canonical event kind, intentional v1 rejection, invalid requests, additive object fields, and unknown-event rejection.

Native Execution Contract `2.0.0` is intentionally breaking. The Rust contract crate owns opaque request parsing, Compiled Transport Problem conversion, engine-result mapping, and expected `runFailed` events. The peer `@transport/native-execution-contract` TypeScript package owns wire types, parsing, compatibility checks, and fixture conformance. No schema or language-binding generator sits between them; the shared reviewable fixture is the parity proof.

## Contract surfaces

The first fixture set should cover:

- compiled photon smoke problem payload
- backend capability metadata
- run-start event
- sampled track event or batch
- tally/update placeholder event if present in the protocol
- diagnostic event
- run-complete event
- run-failed event for bridge-unavailable behavior
- provenance fields needed to explain which backend produced the result

Fixtures should exercise the shape of the protocol rather than claiming physical validation of the simulation result.

## Fixture location

The implementation should choose a location that both TypeScript and Rust tests can read without coupling either side to application internals.

Acceptable options include:

- a neutral `fixtures/bridge/` directory at the repo root
- package-owned fixtures with explicit exports
- generated fixtures committed as JSON with a small schema/version header

The fixture format should be stable, reviewable, and friendly to diffs. JSON is preferred unless a stronger reason emerges.

## Versioning

Every bridge fixture should include an explicit contract version.

Breaking protocol changes update the version and require both TS and Rust tests to move together. The v2 compatibility rule ignores unknown object fields, rejects unknown event kinds, and explicitly rejects v1 requests. Additive metadata therefore remains compatible, while new behavioral event kinds require an intentional contract update.

## Test responsibilities

TypeScript tests should verify:

- compiled payloads match the fixture schema
- native bridge adapters send expected payload shapes
- event stream parsing handles known fixture events
- bridge-unavailable behavior returns the expected diagnostic event
- unknown or unsupported event kinds fail clearly or are ignored only if the protocol permits it

Rust tests should verify:

- native command handlers accept the golden request payload shape
- emitted events match the fixture schema
- diagnostic and failure events remain compatible with the frontend adapter
- protocol version mismatches produce explicit errors

## Relationship to Tauri e2e

These fixtures are not a substitute for eventual Tauri desktop e2e tests.

They are the cheaper first boundary. Tauri e2e should be added later when the native bridge is mature enough that launching the full desktop runtime gives more signal than contract-level fixture checks.

## Non-goals

- Do not add full Tauri desktop automation in this slice.
- Do not require a real native solver to run.
- Do not claim physical correctness from fixture matching.
- Do not make browser-only frontend tests depend on the native runtime.
- Do not move backend protocol ownership into React components.

## Consequences

- TypeScript and Rust can evolve the native bridge with less silent drift.
- Browser-only development remains fast and supported.
- Future Tauri e2e tests can focus on runtime wiring instead of rediscovering basic payload mismatches.
- Fixture changes become explicit review points for protocol compatibility.
- The fixture contract may expose unclear protocol semantics that need a follow-up backend ADR.

## Required follow-up tests

When implemented, add tests for:

- TypeScript compiled-problem-to-native-payload fixture matching
- TypeScript event parser handling of golden native event streams
- TypeScript bridge-unavailable diagnostic fixture matching
- Rust native command request deserialization from the golden payload
- Rust native event serialization against golden event fixtures
- protocol-version mismatch behavior
