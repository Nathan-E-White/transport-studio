# ADR 0001: First Crack Architecture

## Status

Accepted for the prototype phase.

## Context

Transport Studio is intended to support interactive authoring, visualization, validation, and eventual execution of Monte Carlo particle transport problems.

The first implementation needs to prioritize editor ergonomics and visual feedback without prematurely coupling UI state, render state, and physical simulation state. The application should feel like a game-engine editor with a science-CAD flavor, while still leaving room for serious transport backends later.

The prototype also needs a clear boundary between the workbench experience and the transport kernel so that future physics, geometry, tally, material, source, and execution systems can evolve independently.

## Decision

Transport Studio is an editor-first Monte Carlo transport workbench.

The UI owns authoring ergonomics and visualization. The transport engine consumes compiled, validated simulation problems and streams results back to the UI.

The first implementation will favor a clean architectural split over early physical completeness.

## Non-negotiable separations

1. Editable scene is not the simulation problem.
2. Render scene is not the transport geometry.
3. Track visualization is not the tally truth.
4. Data packs are optional assets, not core app dependencies.
5. Backends are plugins behind a stable protocol.

## Conceptual pipeline

```txt
Editable Scene
  -> Validation
  -> Compiled TransportProblem
  -> Backend Protocol
  -> Run Lifecycle / Result Stream
  -> Render Views / Analysis Views / Diagnostics
```

The editable scene may contain incomplete, invalid, decorative, or purely authoring-oriented entities. A transport backend should only receive a compiled and validated `TransportProblem`.

## First backend

The first backend is `transport-visual`: a toy TypeScript photon transport stub optimized for rapid visual feedback.

It is deliberately not the serious physics backend.

Its purpose is to support UI development, interaction design, visual debugging, and early end-to-end workflow testing.

## Future backends

- Web Worker TypeScript backend
- WebGPU backend
- Native Rust/C++ CPU backend
- Native CUDA/HIP/Kokkos/SYCL backend
- Remote/HPC backend

## Backend protocol scope

The backend protocol should eventually cover:

- problem submission
- validation errors and warnings
- run start, pause, resume, cancellation, and completion
- sampled particle tracks
- tally result updates
- diagnostic messages
- backend capability discovery
- backend provenance and run metadata

The exact protocol is deferred to a later ADR, but backend implementations should not depend on React, R3F, editor widgets, viewport state, or UI-specific stores.

## Consequences

- UI entities may exist before they are physically meaningful.
- The app needs an explicit validation and compilation step between editor state and backend execution.
- Translation layers are accepted as the cost of preserving clean boundaries.
- Visualized tracks are diagnostic or illustrative unless tied to explicit backend result provenance.
- Tallies and analysis views should be treated as derived results, not as render artifacts.
- Backend plugins can evolve independently as long as they honor the stable protocol boundary.
- Future tests should distinguish editor behavior, domain validation, backend execution, and visualization behavior.

## UX target

Game-engine editor with science-CAD flavor.

## Follow-ups

- Define the frontend state model for editable scenes, selections, visibility, and mode switching.
- Define the validation and compilation pipeline from editable scene to `TransportProblem`.
- Define the backend protocol and run lifecycle in a dedicated ADR.
- Define result-stream semantics for tracks, tallies, diagnostics, and logs.
- Define how optional material, cross-section, and geometry data packs are discovered and loaded.
