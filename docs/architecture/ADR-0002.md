# ADR 0002: Front-End Beef-Up

## Status

Accepted for the prototype UI shell.

## Context

The initial prototype scaffold established the basic application structure, but it did not yet express the intended workbench experience.

Transport Studio needs to feel like a game-engine-style scientific editor: visual, inspectable, mode-driven, and ready for interactive authoring. At the same time, the UI must not collapse editor affordances into the transport kernel or domain model.

This pass creates visible homes for design, inspection, execution, analysis, and diagnostics before those systems are fully implemented.

## Decision

Create a richer front-end shell that separates mode selection, project navigation, inspection, viewport visualization, and run/result diagnostics while keeping the underlying transport domain model isolated.

This ADR implements the UX direction established in ADR 0001 without changing the core backend boundary.

## Added UI concepts

- Mode switcher: design, probe, run, analyze, debug.
- Project tree with entity summaries, counts, visibility affordances, and kind badges.
- Inspector cards for geometry, materials, sources, and tallies.
- Viewport HUD with active mode, selection, sampled-track counts, and visibility toggles.
- Bottom dock with tabs for run, tallies, tracks, diagnostics, and console.
- Richer R3F scene rendering with source beam guide, tally overlays, labels, event markers, fog, lights, and selection wireframes.

## Still intentionally stubbed

- Entity editing mutators.
- Worker-backed async run loop.
- Real tally accumulation.
- Real cross-section data.
- Native/Tauri compute bridge.

## Architectural constraints

- UI shell state should remain separate from compiled transport problem state.
- Inspector cards may display incomplete or invalid editor entities.
- Viewport rendering may include authoring aids that are not part of transport geometry.
- Bottom-dock run, tally, track, diagnostic, and console views may be backed by mock data during the prototype phase.
- Stubbed UI affordances must not imply that the transport kernel has implemented real physics.

## Consequences

- The UI now exposes affordances for capabilities that are not fully implemented.
- Future implementation work has known visual and interaction targets instead of requiring new layout structure.
- The app can evolve panel-by-panel without forcing premature backend design.
- Tests should verify shell behavior, safe rendering, tab switching, mode switching, and missing-data handling before asserting real physics behavior.
- Domain and backend packages should remain usable without importing React, R3F, or app-shell components.

## Required follow-up tests

- App shell renders without crashing.
- Mode switcher changes active mode.
- Project tree renders geometry, material, source, and tally summaries.
- Inspector handles empty selection.
- Inspector handles each supported entity kind.
- Viewport HUD handles missing sampled tracks.
- Bottom dock tab switching works.
- Run, tally, diagnostic, and console panels tolerate empty or stubbed run data.

## Follow-ups

- Add real editor mutators for geometry, material, source, and tally entities.
- Connect shell state to a deliberate frontend state model.
- Introduce a worker-backed asynchronous run loop.
- Replace visual tally stubs with real result-stream data.
- Connect diagnostics and console views to backend protocol events.
- Define native/Tauri bridge expectations in a later ADR.
