# ADR 0003: Front-End State Model and Editor Domain Boundary

## Status

Accepted for the prototype state model.

## Context

ADR 0001 established that the editable scene is not the simulation problem and that the render scene is not the transport geometry.

ADR 0002 introduced a richer front-end shell with modes, a project tree, inspector cards, a viewport HUD, and a bottom dock for run, tally, track, diagnostic, and console views.

The next architectural gap is the state boundary between the interactive editor and the transport domain model. The UI needs state that supports authoring, selection, visibility, layout, incomplete entities, draft changes, and visual affordances. The transport domain needs clean, validated data structures that can be compiled into a `TransportProblem`.

Without an explicit state model, the app risks leaking React/R3F concerns into the domain packages or, conversely, forcing the editor to pretend that every partially-authored object is already a valid transport object.

## Decision

Introduce an explicit front-end editor state model that is separate from the compiled transport domain model.

The editor state owns authoring-time concerns. The transport domain owns physically meaningful problem definitions. A validation and compilation step converts editable scene state into a clean `TransportProblem` when the user runs or analyzes a problem.

The editor may hold invalid, incomplete, hidden, decorative, locked, selected, draft, or purely visual entities. The compiled transport problem may not.

## State layers

The front end should distinguish at least four state layers:

1. **UI shell state**: active mode, open panels, bottom-dock tab, viewport camera state, hover state, transient drag state, and layout preferences.
2. **Editor scene state**: editable geometry, materials, sources, tallies, labels, annotations, helper objects, selection metadata, visibility flags, lock flags, and dirty flags.
3. **Validation state**: errors, warnings, stale markers, dependency diagnostics, and compile readiness.
4. **Run/result state**: backend run status, progress, sampled tracks, tally updates, diagnostics, logs, provenance, and cancellation status.

These layers may be coordinated by the app, but they should not be collapsed into one untyped bag of state.

Run/result state is owned by a dedicated external Run Session store. React observes stable immutable selections from that store; the editor Context does not own run payloads, and progress-only events do not require whole-workbench subscriptions. Compilation remains outside the store so a compile failure cannot create a session or evict a prior completed result.

## Editor entities

Editor entities should have stable IDs and explicit kinds.

Representative entity kinds include:

- geometry objects
- regions
- surfaces
- materials
- sources
- tallies
- transforms
- annotations
- labels
- visual helpers
- imported assets

Editor entities may include fields that are not physically meaningful, such as display color, collapsed/expanded state, viewport label preference, icon kind, authoring notes, or editor-only tags.

Domain entities should not depend on those fields.

## Selection model

Selection is an editor concern, not a transport-domain concern.

The editor should support:

- no selection
- single selection
- multi-selection
- selection by stable entity ID
- selection by entity kind and ID
- hover/preselection state
- inspector focus independent from viewport hover

Selection must tolerate deleted, missing, hidden, or invalid entities without crashing the inspector or viewport.

## Visibility and locking

Visibility and locking are authoring concerns unless explicitly compiled into a transport problem.

The editor should distinguish:

- visible vs. hidden in the viewport
- selectable vs. non-selectable
- locked vs. editable
- included in compiled problem vs. excluded from compiled problem
- decorative/helper-only vs. domain-significant

A hidden object may still be part of the compiled transport problem. An excluded object may still be visible in the editor. A visual helper may be visible but never compiled.

## Dirty and stale state

The editor should track whether derived state is stale.

Examples:

- geometry changed after validation
- material assignment changed after tally configuration
- source changed after sampled tracks were generated
- backend results exist for an older compiled problem
- diagnostics no longer match the current editor scene

When the editor scene changes, run results and validation results may become stale rather than immediately deleted. Current-scene rendering must block stale tracks and tallies. The UI may render them only by switching to the lightweight submitted-scene snapshot preserved by the Run Input Record, while explaining the submitted and current revisions.

## Boundary with render state

R3F/Three.js objects are render artifacts and should not be treated as source-of-truth domain entities.

The render scene may contain:

- meshes
- lights
- fog
- labels
- helper grids
- selection outlines
- source beam guides
- tally overlays
- track polylines
- event markers
- camera controls

These may be generated from editor state, run/result state, or static visual defaults. They should not be required to construct a `TransportProblem`.

## Boundary with transport domain

The transport domain should receive only validated, compiled data.

The compiled `TransportProblem` should contain domain-significant objects such as:

- validated geometry
- region and surface definitions
- material definitions and assignments
- source definitions
- tally definitions
- boundary conditions
- run controls
- references to approved data packs or backend capabilities

It should not contain React component state, R3F objects, inspector UI state, panel layout, viewport camera state, hover state, collapsed tree nodes, or editor-only decorations.

## Compilation rule

The compilation pipeline is the only approved path from editable scene state to backend execution.

```txt
EditorSceneState
  -> validateEditorScene(...)
  -> compileTransportProblem(...)
  -> TransportProblem
  -> BackendProtocol.submit(...)
```

Validation may produce errors and warnings. Errors block compilation. Warnings may allow compilation depending on the backend and run mode.

## Consequences

- The editor can support rich authoring workflows without pretending every draft object is physically valid.
- The transport domain remains usable outside the React app.
- Backends remain insulated from UI and render concerns.
- Tests can separately exercise UI state, editor scene mutation, validation, compilation, rendering, and backend execution.
- The app needs explicit adapters between editor entities and domain entities.
- Some duplication is acceptable when editor-facing objects and domain-facing objects serve different purposes.

## Required follow-up tests

- Selection handles empty, missing, hidden, and deleted entities safely.
- Visibility toggles do not imply compile inclusion changes unless explicitly requested.
- Compile inclusion can differ from viewport visibility.
- Editor-only helper entities are never emitted into `TransportProblem`.
- Dirty/stale flags update when geometry, materials, sources, or tallies change.
- Validation errors block compilation.
- Validation warnings can be surfaced without crashing the run workflow.
- Run results can be marked stale after editor changes.

## Follow-ups

- Implement an editor-state module for mode, selection, visibility, locking, dirty/stale markers, and project-tree summaries.
- Define `EditorSceneState` and editor entity metadata types.
- Define validation result types shared by the editor and domain packages.
- Define the first `compileTransportProblem(...)` adapter.
- Add tests for editor-state reducers or store actions.
- Add tests proving that domain packages do not import React, R3F, or UI-shell modules.
